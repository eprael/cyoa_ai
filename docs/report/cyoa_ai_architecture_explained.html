CYOA AI — Architecture Explained

How user story requests flow from browser through a job queue to AI services and back.

This document explains the async job queue pattern used by the CYOA AI application. Rather than making users wait while AI APIs process requests, the app logs jobs immediately, processes them in the background, and polls for results — keeping the interface fast and the architecture clean.

The Five Panels

🟠 Browser

Client-side

This represents the user's web browser. Two distinct things happen here:

  • Generate Story button — the user clicks this to kick off the whole flow. It triggers a single HTTP request to the PHP server.
  • Polling loop (every 4 seconds) — a JavaScript timer runs independently, silently calling the server to check if the job is done. When the result is ready, it displays it. The clock icon signals this is time-driven — it fires on a schedule, not in direct response to a user action.

🔵 Web Application (PHP)

Server-side

The PHP server the browser talks to directly. It handles two lightweight responsibilities:

  • Insert row / Set status to pending — when the button is clicked, PHP immediately writes a job record to the database and returns. It does not call the AI API here. It's just taking the order.
  • Get job status — when the browser polls every 4 seconds, PHP performs a simple database read and returns the current status. This is intentionally lightweight.

🟢 Job Queue Table (mySQL)

Shared State

The heart of the async pattern — a MySQL table called ai_jobs. Every AI request gets a row here. The web app and the cron worker never talk to each other directly; they only read and write this table.

🟣 Cron Worker

Background

A background PHP script triggered by the server's cron scheduler every 30 seconds. Users never interact with it directly. It follows five steps:

  • Pick oldest pending job — queries for rows with status = pending
  • Mark as running — updates immediately so no other worker instance picks the same job
  • Call AI API — sends the prompt to Claude or DALL-E depending on job type
  • Store result — writes the AI response back to the database row
  • Mark as completed — so the browser's next poll finds it

🔵 Cloud (AI Services)

External APIs

The external AI APIs the cron worker calls out to:

  • 🧠 Claude — handles text-based requests: scenes, stories, dialogue
  • 🖼️ DALL-E — handles image generation requests

The worker sends a JSON payload and receives a JSON response. These are third-party services with variable response times — the reason the async queue exists in the first place.

End-to-End Request Flow

Request lifecycle
01 User clicks Generate Story in the browser Browser
02 Browser sends HTTP request to the PHP web app Browser → PHP
03 PHP inserts a row into ai_jobs with status = pending and returns immediately PHP → DB
04 Every 4 seconds, the browser polls PHP: "is my job done?" Browser → PHP → DB
05 Every 30 seconds, the cron worker wakes up and finds the pending job Cron → DB
06 Worker marks job running, calls the appropriate AI API Cron → Cloud
07 AI API returns result; worker stores it and marks job completed Cloud → Cron → DB
08 Browser's next poll finds status = completed and retrieves the result Browser → PHP → DB
09 Browser displays the generated story or image to the user Browser

Why This Pattern?

AI APIs can take anywhere from a few seconds to over a minute to respond. Calling them synchronously during a user's HTTP request would cause the browser to hang — and most web servers time out after 30–60 seconds anyway. The job queue pattern solves this cleanly:

⚡

Fast web server

PHP only does lightweight DB reads and writes — it never blocks waiting for an AI response.

🔄

Decoupled processing

The cron worker runs independently. A slow or failed AI call doesn't affect the web app at all.

🧾

Durable record

Every request is logged in the database — easy to inspect, retry, audit, or debug.

🧑‍💻

Smooth UX

Users see a progress indicator rather than a frozen page or browser timeout error.

This is a well-established pattern in web development — sometimes called a background job queue or task queue. The same underlying concept is used by:
Laravel Queues Sidekiq (Ruby) Celery (Python) Bull (Node.js) Resque Amazon SQS

The CYOA implementation keeps it simple: a MySQL table and a cron job — no external queue infrastructure required.
